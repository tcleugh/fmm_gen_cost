# Reviewer's Guide

This branch is large (+18k) because it bundles four logically distinct streams of work into a single MR. Most of the diff is framework boilerplate, design documents, or vendored fixtures — the substantive C++ surface is closer to ~9k of new code spread across three features.

This guide tells you which paths matter, which can be skimmed, and which can be skipped entirely.

## Quickest path: review by commit

The branch is 25 commits; each commit is a self-contained logical unit and reviewing commit-by-commit is far more digestible than the rolled-up diff. Recommended order:

```bash
git log --oneline master..HEAD --reverse
```

If you want to validate end-to-end correctness in one command:

```bash
mkdir -p build && cd build && cmake .. && make polymatch_test -j$(nproc)
./polymatch_test                     # 55 cases, 2868 assertions
./polymatch_test '[real_network]'   # 11 cases on real shapefile data
```

Both should print `All tests passed`. The real_network ledger shows all four invariants at 200/0 (and 20/0 for link-only ≡ weightmatch) — that's the post-003 steady state.

## Useful invariants to spot-check while reviewing the matcher

Any change to `build_hybrid_path` (`src/mm/polymatch/polymatch_algorithm.cpp`) MUST preserve these:

1. **cpath-topology**: every adjacent `(polygon, edge)` or `(edge, polygon)` pair in cpath has the edge AP-incident to the polygon.
2. **is-through-has-aps**: every `PolygonSegment` with `is_through == true` has both `entry_ap` and `egress_ap` set.
3. **link-only-eq-weightmatch**: trips with no polygon Viterbi candidates produce cpath byte-identical to weightmatch.
4. **distance-inside-finite**: `distance_inside` is finite and non-negative for every polygon segment.

These four invariants are encoded in `test/polymatch_test.cpp` as `check_*` predicates feeding a `ViolationLedger`. If you change anything in the matcher and don't see the predicates fire, the change is silently broken.
